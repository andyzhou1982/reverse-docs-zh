# 详细设计逆向生成器

## 模板文件

**在开始任何工作之前，必须先读取模板文件：**

```
references/detailed-design-template.md
```

生成的文档必须严格遵循该模板的章节结构。模板中标注 `[重要]` 的章节，是详细设计文档的重点；模板中标注 `[可选]` 的章节，仅在代码中能找到相关信息时才生成；模板中标注 `[待填写]` 的占位符，必须替换为从代码中提取的实际内容，无法填充的章节直接跳过。

## 核心原则

**绝不编造。** 文档中的每一项内容都必须来自代码知识图谱或实际源代码的验证。如果模板中某个章节无法从代码中获取足够信息，则跳过该章节，绝不允许凭空填充。

## 代码分析策略：GitNexus 优先，降级兜底

本 skill 使用**双层探测策略**理解代码结构：

### 优先模式：GitNexus 知识图谱

如果项目已接入 GitNexus MCP，优先使用知识图谱工具。图谱预先构建了完整的符号关系、调用链路和执行流程，能高效回答以下问题：

| 需要了解的 | GitNexus 工具 | 作用 |
|-----------|---------------|------|
| 项目有哪些功能域 | `READ gitnexus://repo/{name}/clusters` | 功能集群及内聚度 |
| 代码有哪些执行流程 | `READ gitnexus://repo/{name}/processes` | 预计算的调用链 |
| 某概念的代码在哪 | `gitnexus_query({query: "关键词"})` | 按相关性排名的流程和符号 |
| 某符号的完整关系 | `gitnexus_context({name: "符号名"})` | 360° 视图：调用者、被调用者、参与的流程 |
| 修改某符号的影响 | `gitnexus_impact({target: "符号名"})` | 影响半径和风险等级 |

**判断 GitNexus 是否可用**：尝试调用 `gitnexus_list_repos`。如果调用成功且返回了当前项目的索引，则使用 GitNexus 优先模式；如果调用失败或未找到当前项目，则进入降级模式。

### 降级模式：Claude Code 内置工具

当 GitNexus 不可用时，使用 Claude Code 自带的工具手动分析代码结构：

| 需要了解的 | 工具 | 用法 |
|-----------|------|------|
| 项目有哪些文件 | Glob | `Glob("**/*.java")` 或 `Glob("**/views/*.py")` |
| 谁调用了某函数/类 | Grep | `Grep("import.*UserService")` 找依赖 |
| 某接口的所有实现 | Grep | `Grep("implements.*UserService")` 或 `Grep("class.*\\(.*UserService\\)")` |
| 代码中的调用关系 | Grep + Read | 先 grep 找调用点，再 read 阅读上下文确认 |
| 函数/方法定义 | Read | 读取源文件提取签名和注释 |
| 目录结构 | Bash | `ls`、`find` 列出目录和文件 |

**降级模式下分析调用关系的技巧：**
1. 从入口文件（main、router、app）开始，逐层 trace 调用链
2. 用 Grep 搜索 import 语句确定模块间依赖
3. 用 Grep 搜索函数/方法名确定调用者和被调用者
4. 用 Read 读取关键文件的完整内容确认调用关系
5. 可以用 Agent 工具并行派出多个子 agent 分别分析不同模块

## 阶段零：识别技术栈

在开始收集信息前，先快速判断项目的技术栈，这决定了后续探索的策略。

**识别信号：**

| 文件/目录 | 技术栈 |
|-----------|--------|
| `pom.xml` / `build.gradle` | Java (Maven/Gradle) |
| `requirements.txt` / `setup.py` / `pyproject.toml` | Python |
| `go.mod` | Go |
| `package.json` / `node_modules` | Node.js / 前端 |
| `Cargo.toml` | Rust |
| `*.csproj` / `*.sln` | .NET / C# |

项目可能包含多个技术栈（如 monorepo 中前后端不同语言）。对每个模块分别识别。

## 工作流程

### 阶段一：收集模板与项目概况

并行执行以下操作：

1. **读取模板文件** — 必须读取 `references/detailed-design-template.md`，这是文档结构的唯一依据。如果用户提供了其他模板路径，以用户提供的为准
2. **探测 GitNexus 可用性** — 调用 `gitnexus_list_repos`：
   - **成功且包含当前项目** → 进入 GitNexus 优先模式，继续以下步骤 3
   - **失败或未找到当前项目** → 进入降级模式，跳过步骤 3，所有信息收集改用 Glob/Grep/Read
3. **获取知识图谱概览**（仅 GitNexus 可用时）：
   - `READ gitnexus://repo/{name}/context` — 仓库概览与统计
   - `READ gitnexus://repo/{name}/clusters` — 功能集群列表（直接反映代码的功能域划分）
   - `READ gitnexus://repo/{name}/processes` — 执行流程列表
4. **探索目录结构** — 用 Bash 列出顶层模块目录（两种模式都需要）
5. **读取 CLAUDE.md**（如果存在）— 项目约定的架构说明和分层规则

### 阶段二：深入探索各模块

对每个模块，并行收集以下信息。**根据技术栈选择对应的探索策略：**

#### 通用探索（所有语言）

- 用 Bash 列出模块的源码目录结构
- 读取依赖配置文件获取版本与外部依赖（pom.xml / package.json / go.mod / requirements.txt / pyproject.toml）
- 读取 Dockerfile（如有）获取部署信息
- 读取配置文件（application.yml / .env / config.yaml / settings.py / config.json 等）获取端口、中间件、安全等配置
- 查询该模块的核心业务流程：
  - **GitNexus 模式**：`gitnexus_query({query: "业务关键词", repo: "项目名"})`
  - **降级模式**：用 Grep 搜索路由/接口定义关键词（如 `@RequestMapping`、`router.get`、`func.*Handler`），再用 Read 读取关键文件理解业务逻辑

#### 各语言探索策略

根据识别到的技术栈，读取 `references/language-exploration-guide.md` 中对应的语言章节，应用匹配的文件搜索模式和探索策略。

#### 其他语言

使用通用模式：找到项目的入口文件、路由/API 定义、数据模型定义、外部调用点、配置文件。知识图谱查询是语言无关的，可以补充任何语言的代码结构理解。

### 阶段三：交叉验证与补充

1. 查询横切关注点：
   - **GitNexus 模式**：`gitnexus_query` 查询安全/认证、外部系统集成、消息队列等
   - **降级模式**：用 Grep 搜索关键词（如 `auth`、`middleware`、`client`、`producer`、`queue`），再用 Read 读取匹配文件确认

2. 对关键实体（核心数据模型、核心业务对象）用 Read 工具读取完整定义，提取字段名、类型、注释

3. 读取安全/中间件配置文件（auth 配置、网关配置等）

### 阶段四：生成文档

按照模板结构组织文档内容。各章节的信息来源和处理方式根据实际发现的技术栈灵活适配：

| 模板章节 | 信息来源 | 处理方式 |
|----------|----------|----------|
| 概述/术语 | CLAUDE.md + 代码注释 + 知识图谱 | 直接整理 |
| 系统上下文 | 外部调用客户端列表 + 消息生产者 + 配置文件 | 从代码中提取外部系统名 |
| 整体架构 | 模块目录结构 + CLAUDE.md 依赖关系 + 知识图谱 | 生成 Mermaid 架构图 |
| 技术选型 | 依赖配置文件 + 配置文件 | 提取版本号、框架名 |
| 部署架构 | Dockerfile + 配置文件 + CI 配置 | 提取镜像、端口、环境变量 |
| 模块详细设计 | API 层列表 + 数据模型列表 + 知识图谱 | 按模块逐一描述 |
| 核心流程 | gitnexus_query + 知识图谱 processes | 生成 Mermaid 时序图/状态图 |
| 数据设计 | 数据模型定义文件（ORM/Schema/Entity） | 提取表名和字段 |
| 接口设计 | 模块间调用关系（RPC/HTTP/MQ） | 整理调用关系表 |
| 安全设计 | 安全配置文件 + 中间件 | 提取认证方案 |
| 可靠性设计 | 配置文件中的熔断/限流/重试/MQ 配置 | 提取配置参数 |

**信息不足时的处理：**
- 如果模板某章节无法从代码中获取足够信息，则**跳过该章节**
- 在文档末尾注明哪些章节因信息不足被跳过
- 不要用"待填写"占位符——直接不写该章节

### 阶段五：自检

生成文档后，执行以下检查：

1. 文档中提到的每个模块名、类名/文件名是否在代码中实际存在
2. 文档中提到的每个外部系统是否在代码中有对应的调用点
3. 架构图中的依赖方向是否符合实际代码的依赖关系
4. 数据模型的字段是否来自实际读取的源文件
5. 状态枚举值是否与代码中的定义一致

如果任何检查项失败，修正文档中的不准确内容后重新自检，直到全部通过。

## 文档编写规范

### 语言
- 使用中文编写文档
- 技术术语保留英文原文（如 REST、gRPC、Redis、PostgreSQL）

### 图表
- 架构图使用 Mermaid `graph` 语法
- 时序图使用 Mermaid `sequenceDiagram` 语法
- 状态图使用 Mermaid `stateDiagram-v2` 语法
- ER 图使用 Mermaid `graph LR` 语法

### 模块描述格式
对每个模块使用统一的职责表：

```markdown
| 项目 | 说明 |
|------|------|
| 模块名称 | [名称] |
| 所属层次 | [层级] |
| 核心职责 | [一句话描述] |
| 依赖模块 | [依赖列表] |
```

### 数据收集效率

- **尽量并行**：阶段一和阶段二的多个操作应使用并行工具调用
- **分层收集**：先收集宏观信息（目录、列表），再有针对性地读取关键文件
- **避免冗余**：对于大型模块，列出文件名即可，不必逐个读取所有文件
- **重点深入**：只对核心数据模型读取完整字段定义，其余列出名称即可
- **知识图谱优先，降级兜底**：GitNexus 可用时用它获取调用关系和执行流程；不可用时用 Grep/Read 手动 trace，效果相同但耗时更长
- **降级模式可多用 Agent**：没有知识图谱时，可以用 Agent 工具并行派出子 agent 分别分析不同模块，提高效率
